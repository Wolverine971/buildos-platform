# BuildOS Demo Batch Recording Checklist

## Goal

Batch-record the first two demos in one session:

- Weekly video essay pipeline
- Fantasy novel revision

Keep the whole recording session under 45 minutes.

---

## Before You Record

### Product prep

- create one clean YouTube example project
- create one clean author example project
- hide any confusing or unfinished internal-only data
- make sure the "next move" task is visible in both projects
- confirm the relevant docs open quickly

### Screen prep

- clean desktop
- close unrelated tabs
- use one browser window only
- zoom UI to mobile-readable size
- turn off distracting notifications
- set browser zoom to a consistent level for both demos

### Recording prep

- decide horizontal or vertical first
- if horizontal: record at 1200px+ width
- if vertical: keep the focal UI centered so it crops well
- record cursor movement slowly
- pause one beat after every important state change

---

## Assets To Have Ready

### Demo 1: YouTube workflow

Raw dump:

```text
Need a cold open for the AI fatigue video, stronger examples, sponsor slot, B-roll list, and three shorts once the main edit is done.
```

Key states to show:

- raw dump input
- project created
- research doc
- production checklist
- next move: draft the cold open

### Demo 2: Author workflow

Raw dump:

```text
Chapter 9 drags. The queen's motive still feels weak. I need a magic-rules doc, and I keep forgetting which clues I planted in act 1.
```

Key states to show:

- raw dump input
- project created
- clue tracker
- character note or world-rules doc
- next move: rewrite chapter 9

---

## Recording Order

### Pass 1: Capture clean product footage only

Do not talk. Just capture screen movement.

For each demo record:

1. hook state
2. raw input paste
3. transformation state
4. one context view
5. next move state

### Pass 2: Record voiceover separately

This gives you:

- cleaner timing
- easier retakes
- easier reuse across platforms

### Pass 3: Add burned-in captions

Keep captions short.
Do not caption every spoken word if it hurts readability.

---

## Timing Template

### Recommended structure for each demo

- `0:00-0:03` hook
- `0:03-0:08` raw input
- `0:08-0:15` transformation
- `0:15-0:23` open structure/context
- `0:23-0:30` highlight next move
- `0:30-0:33` close screen / logo / stable state

Target:

- 25-35 seconds total

---

## Naming Convention

Use simple filenames so assets do not get lost.

### Raw captures

- `2026-03-09_demo-youtube-workflow_take-01.mov`
- `2026-03-09_demo-author-revision_take-01.mov`

### Voiceovers

- `2026-03-09_vo-youtube-workflow_take-01.wav`
- `2026-03-09_vo-author-revision_take-01.wav`

### Exports

- `2026-03-09_buildos-thinking-environment-youtube-demo_v1.mp4`
- `2026-03-09_buildos-thinking-environment-author-demo_v1.mp4`

---

## Caption Rules

- one idea per caption
- 3 lines max
- avoid dense full-sentence blocks
- keep the key transformation visible:
  raw input -> project -> next move

Good:

- `raw creator input`
- `BuildOS turns it into a real project`
- `next move: draft the cold open`

Bad:

- long explanatory text that competes with the interface

---

## Export Settings

### Horizontal

- 1920x1080
- H.264
- high bitrate but web-friendly

### Vertical cutdown

- 1080x1920
- crop around the active UI zone
- keep text centered and large

---

## Quality Check Before Posting

- can the raw input be read on mobile
- is the transformation obvious without audio
- is the next move visible
- does the video avoid generic AI language
- does it feel like a thinking environment, not a dashboard tour

If not, simplify the edit.

---

## One-Session Recording Plan

### 0-10 min

- clean desktop
- open both prepared projects
- test zoom level

### 10-20 min

- capture YouTube demo footage

### 20-30 min

- capture author demo footage

### 30-40 min

- record both voiceovers

### 40-45 min

- quick review and identify best takes

---

## Bottom Line

The demo does not need to explain everything.

It only needs to prove one thing:

**BuildOS takes messy creator input and turns it into structured work with a visible next move.**
