# Do This Asset Template

## Purpose

Use this format when you want to make one sharp comparison without sounding negative for the sake of being negative.

This format works because it:

- gives a clear prescription
- still creates contrast
- feels useful, not preachy
- translates well to X, LinkedIn, and Instagram

---

## Recommended Format

### Title

Use:

**WHEN [AUDIENCE] [PAIN STATE]**

Examples:

- WHEN YOUR PROJECT LIVES IN FIVE PLACES
- WHEN CONTENT CREATION FEELS HEAVIER THAN IT SHOULD
- WHEN YOU KEEP REBUILDING THE SAME PROJECT
- WHEN YOUR WORKFLOW IS HOLDING TOO MUCH IN YOUR HEAD

### Column Headers

Recommended:

- `DO THIS`
- `INSTEAD OF THIS`

If you want softer language:

- `DO THIS`
- `SKIP THIS`

Do not use:

- `RIGHT / WRONG`
- `GOOD / BAD`

Those feel too preachy.

---

## Copy Structure

Use 3-4 bullets max per column.

The bullets should be short enough to read in one glance.

### Left Column: DO THIS

Put the better behavior here.

Template:

- [simple action]
- [system behavior]
- [focus action]
- [memory/restart benefit]

### Right Column: INSTEAD OF THIS

Put the old broken behavior here.

Template:

- [fragmented behavior]
- [holding context in your head]
- [rebuilding from scratch]
- [tool-sprawl behavior]

---

## First Recommended BuildOS Version

### Title

**WHEN YOUR PROJECT LIVES IN FIVE PLACES**

### Left Column

**DO THIS**

- Keep the project in one place
- Attach notes, tasks, and AI context to the same project
- Leave yourself one clear next move
- Return to the work without rebuilding everything

### Right Column

**INSTEAD OF THIS**

- Bouncing between notes, docs, and chat threads
- Keeping the plan in your head
- Reopening tabs to remember what you meant
- Restarting the project every time you sit down

---

## Alternate Version For Creators

### Title

**WHEN CONTENT CREATION FEELS HEAVIER THAN IT SHOULD**

### Left Column

**DO THIS**

- Keep research, hooks, script notes, and next steps in one project
- Let the work stay connected as it evolves
- Make the next move visible before you leave
- Come back to context, not chaos

### Right Column

**INSTEAD OF THIS**

- Notes app for ideas
- Google Doc for the draft
- AI chat for the good thinking you forget to save
- Checklist somewhere else

## Design Template

### Layout

- 4:5 portrait canvas
- centered title in upper third
- two columns below
- left column slightly emphasized
- thin vertical divider line
- generous spacing between bullets

### Text Hierarchy

- title largest
- column headers small but strong
- bullets readable at phone size

### Best Font Direction

- title: editorial serif or tasteful display serif
- body: readable serif or clean sans

The tone should feel like:

- calm
- intelligent
- intentional

Not:

- corporate SaaS
- flashy startup ad

---

## Background Recommendation

### Best Option

Use a **darkened working-surface background**.

Recommended background subjects:

- overhead desk with paper, notes, and a keyboard
- writing table with scattered notes
- editorial wall or project map
- blurred workspace with real working texture

Why this works:

- it matches the "thinking environment" idea
- it feels like real work
- it supports the contrast between chaos and structure
- it is more on-brand than scenery

### Background Treatment

- slightly desaturated
- blurred or softened
- dark overlay around 45-60%
- enough texture to feel alive
- not so much detail that it fights the copy

### If You Need A Cleaner Version

Use:

- textured slate background
- warm charcoal gradient
- paper-grain background with subtle vignette

That is better than generic stock photos.

### Avoid

- nature scenery unless it directly supports the message
- generic office stock imagery
- bright busy screenshots behind text
- heavy purple gradients
- anything that feels mystical, futuristic, or robot-branded

For BuildOS, the background should feel like:

**a project war room, writing table, or working desk**

not:

**a landscape poster**

## Image Treatment Options

The best `DO THIS / INSTEAD OF THIS` assets usually use one of these three approaches.

### Option 1: Editorial Text Card

Use this first if you want speed.

What it looks like:

- strong title at top
- two clean text columns
- textured dark workspace background
- no extra screenshots inside the card

Why it works:

- fastest to make
- cleanest at phone size
- feels intentional

Use this when:

- the message matters more than feature proof
- you want a repeatable series format

### Option 2: Fragmented Vs Unified Visual

This is the best option for BuildOS if you have time.

What it looks like:

- left side shows fragmented pieces
- right side shows one unified BuildOS project
- title above
- text columns below or overlaid lightly

Visual idea:

- `INSTEAD OF THIS` side: small stacked windows, notes, docs, chat bubbles, checklist fragments
- `DO THIS` side: one calm project view with structure and one visible next move

Why it works:

- communicates replacement instantly
- makes the product feel real
- supports the category without heavy explanation

Use this when:

- you want the asset to do both positioning and proof

### Option 3: Screenshot-Led Card

Use this when the actual BuildOS screen looks strong enough.

What it looks like:

- title at top
- one featured BuildOS screenshot anchoring the design
- smaller supporting fragments or labels around it
- copy kept shorter

Why it works:

- strongest product proof
- useful once the UI looks visually coherent

Use this when:

- you have a screenshot that already looks calm and legible

---

## What The Images Should Actually Show

For BuildOS, the visual should show:

### `DO THIS`

- one project home
- connected notes, tasks, and documents
- one visible next step
- visual calm
- fewer objects
- stronger hierarchy

### `INSTEAD OF THIS`

- scattered notes
- separate doc fragments
- chat bubbles or AI-thread fragments
- checklist fragments
- tab overload or visual clutter
- weak hierarchy

The left side should feel:

- calmer
- more decisive
- more usable

The right side should feel:

- fragmented
- mentally expensive
- familiar in a slightly painful way

---

## Best First Version To Make

For day one, make this:

- top title
- two-column layout
- darkened desk or writing-table background
- small fragmented overlays behind `INSTEAD OF THIS`
- one clean BuildOS screenshot or mock panel behind `DO THIS`

That gives you:

- contrast
- product proof
- enough clarity to post immediately

If production feels heavy, drop the overlays and make the editorial text-card version first.

---

## Asset Rule

If someone only looks at this card for two seconds, they should understand:

- what habit you are advocating
- what broken pattern you are replacing
- that BuildOS is about a better working system

If it only looks pretty, it failed.
