# Creator Acquisition Operating System

**Status:** current operating plan — 2026-07-10
**Owner:** DJ Wayne
**First campaign:** 30-day Writer / Cold Manuscript pilot
**Second campaign:** YouTuber / Channel Memory pilot, after the Writer review

## Purpose

This folder turns the BuildOS worldbuilding work into an acquisition and activation loop.

The job is not to publish more ideas. The job is to:

1. enter the conversations where working creators already describe their problems;
2. turn those problems into real BuildOS proof;
3. distribute one proof asset in three platform-native forms;
4. move interested creators into an audience-specific starting path;
5. measure whether they reach the remembered-project moment and return.

## Decisions Locked

1. **One brand, two audience dialects.** BuildOS remains a thinking environment for people making complex things. Writer and YouTuber language are campaign translations, not new company categories.
2. **Writer first.** Run one coherent 30-day Writer pilot. Do not alternate Writer and YouTuber posts day by day. Run the YouTuber pilot second.
3. **DJ distributes; BuildOS proves.** Personal accounts carry conversation, judgment, and founder trust. BuildOS accounts/pages act as proof libraries and conversion hubs.
4. **One weekly receipt.** Every campaign week begins with a real project, screenshot, screen recording, creator session, or product decision. No receipt means no campaign asset yet.
5. **Real media by default.** Use real product screens, screen recordings, founder footage, real workspace photography, and Inkprint explanatory graphics. Photoreal AI scenes are not proof and stay off by default.
6. **Research must create action.** A watering-hole session is incomplete until it produces at least one executed touch, creator invite, language capture, or content experiment.
7. **Social does not terminate at the generic homepage.** Writer and YouTuber traffic need different proof, entry prompts, and attribution even if they share the same product.
8. **Activation outranks reach.** Qualified conversations, real first projects, and seven-day returns matter more than follower totals.

## Acquisition Loop

```text
creator conversation
  -> named pain in their language
  -> real BuildOS transformation
  -> platform-native post
  -> audience-specific landing / entry path
  -> first real capture
  -> useful project structure
  -> return-to-project moment
```

## Account And Channel Jobs

| Surface   | Primary account/job                                           | Supporting account/job                                          | Minimum weekly output                             | Primary signal                                         |
| --------- | ------------------------------------------------------------- | --------------------------------------------------------------- | ------------------------------------------------- | ------------------------------------------------------ |
| Instagram | DJ: discovery, creator relationships, visual story            | BuildOS: Collab partner, proof grid, product credibility        | 1 Reel or carousel adaptation + 10 useful touches | Shares/sends, non-follower reach, qualified DMs        |
| LinkedIn  | DJ: founder trust, category explanation, creator case studies | BuildOS Page: credibility hub, proof archive, selective reposts | 1 Receipt Post + 5 useful comments                | Qualified comments, profile views, demo/landing visits |
| X         | DJ: audience listening, hook testing, direct conversation     | BuildOS: release/proof support only                             | 2 original tests + 10 useful replies              | Qualified replies, profile visits, DMs/clicks          |

The minimum is intentionally lower than older 5-post-per-week plans. Execution quality and proof are the constraints.

## Content Architecture

The existing content lanes are sufficient. Worldbuilding supplies audience language inside them.

| Existing lane      | Writer expression                              | YouTuber expression                                             |
| ------------------ | ---------------------------------------------- | --------------------------------------------------------------- |
| Demonstration      | Resume a cold manuscript; recover open threads | Move a spark into a ripe video queue; create an editor handoff  |
| Worldview          | Re-entry is often the real block               | Output mode can consume input mode                              |
| Founder proof      | BuildOS organizes a real owned writing project | BuildOS organizes its own content operation                     |
| Audience resonance | Drawer novel, critique pile, story-bible drift | Idea graveyard, upload gap, analytics shame, tiny-team handoffs |

Public concept limit:

- Writer: **re-entry tax**, **Manuscript Memory**, **cold manuscript / story stays warm**.
- YouTuber: **Channel Memory**, **Ripe Queue**, **Input Mode / Output Mode**.

Everything else stays supporting language until creators repeat it back.

## Activation Definition For This Program

**Initial activation**

A creator submits a real project capture, receives structure they recognize as useful, and accepts, edits, or acts on part of it.

**Durable activation**

The creator returns within seven days and can recover the project's state or next move without reconstructing it from scratch.

Do not treat registration, onboarding completion, or a generated demo project as activation.

## Weekly Operating Rhythm

| Day       | Work                                                                                        |
| --------- | ------------------------------------------------------------------------------------------- |
| Monday    | Mine 3–5 live audience conversations; execute at least 2 touches; select the week's receipt |
| Tuesday   | Capture the real screenshot/screen recording; draft the LinkedIn Receipt Post               |
| Wednesday | Publish LinkedIn; remain available for replies; cut the IG version                          |
| Thursday  | Publish or Trial the IG asset; run creator setup/interview calls                            |
| Friday    | Publish 1–2 X hook/proof tests; complete the scorecard; choose keep/iterate/kill            |

This is a default sequence, not a requirement to wait until Wednesday to ship a perishable post.

## Required Decisions Before Public Launch

- [ ] Choose the LinkedIn identity mode: founder-forward, recruiter-forward, or a deliberate hybrid. The pending recruiter-first rewrite conflicts with BuildOS acquisition if pasted unchanged.
- [ ] Confirm DJ personal accounts are the distribution engine and BuildOS accounts are the proof/conversion layer.
- [ ] Confirm the strict real-media rule for this pilot: no photoreal AI desk scenes. If an exception is used, make it visibly illustrative and label it as a deviation.
- [ ] Choose the Writer destination for the first campaign: a dedicated route, a focused public page, or a temporary audience-specific onboarding link.
- [ ] Define what the current product can truthfully demonstrate. Do not publish `Memory Mode`, `story bible builds itself`, or `resume in five minutes` as present-tense capabilities until the recorded product proves them.
