# Creator Receipt Kit

**Purpose:** produce one real proof asset each week and adapt it without creating three separate content jobs.

## Media Decision

Default for the Writer pilot:

- real BuildOS screenshots;
- real BuildOS screen recordings;
- real footage/photos of DJ and his workspace;
- Inkprint text, diagram, and annotation layers;
- creator footage or artifacts only with explicit permission.

Photoreal AI desk scenes are off for this pilot. If DJ later opens the abstract AI-scene exception, the output must be visibly illustrative, never presented as real proof, and logged as a brand deviation. Product UI and people are never synthesized.

## The Weekly Receipt Bundle

Each bundle begins with one project moment and creates:

1. a master screen recording or before/after capture;
2. a clean product screenshot;
3. an annotated screenshot;
4. one vertical 9:16 cut for Instagram;
5. one LinkedIn-native image/document or short clip;
6. one X image/short clip;
7. a proof note recording exactly what happened, what was manual, and what remains imperfect.

## One-Hour Real Workspace Shoot

Capture once; reuse as supporting texture for the full campaign.

### Surfaces

- [ ] Clean top-down desk, empty center
- [ ] Messy top-down desk with notebook, index cards, phone, and printed notes
- [ ] Laptop open on a real BuildOS screen
- [ ] Printed BuildOS screenshot on paper
- [ ] Notebook beside laptop with one handwritten next move

### Human Detail

- [ ] Hands typing
- [ ] Hand moving an index card
- [ ] Hand marking a printed screenshot
- [ ] Closing the laptop
- [ ] Returning with keys/coffee and reopening it

### Texture

- [ ] Paper edge / tape / marker close-up
- [ ] Notebook margin and handwriting
- [ ] Scratched desk or actual work surface
- [ ] Phone voice memo or notes screen beside BuildOS
- [ ] Wide environmental shot for profile/banner use

Capture both horizontal and vertical versions where practical. Avoid personal data, notifications, browser tabs, or client material in frame.

## Screen-Recording Standards

Before recording:

- [ ] Use a real owned project or permissioned creator project.
- [ ] Remove or blur private names, emails, calendars, and client information.
- [ ] Close unrelated tabs and notifications.
- [ ] Set the browser zoom so the meaningful state is readable on a phone.
- [ ] Identify the single transformation being shown.
- [ ] Write down the exact start state and end state.
- [ ] Decide which steps are manual so the edit cannot imply automation that did not occur.

Record:

- [ ] A clean master take with natural cursor movement.
- [ ] A second take with pauses at the hook, transformation, and next move.
- [ ] A still screenshot of the before state.
- [ ] A still screenshot of the after state.
- [ ] A close crop of the proof point.

After recording:

- [ ] Verify all displayed claims against what the recording shows.
- [ ] Add captions that remain readable with sound off.
- [ ] Remove dead time without making the transformation look instantaneous if it was not.
- [ ] Label internal/owned examples as such.
- [ ] Record the date, project, permissions, and product version.

## Four Reusable Asset Templates

### 1. The Mess

Use for the audience-recognition hook.

- Real source material: notes, voice transcript, files, feedback, index cards.
- Inkprint `Static` texture only as an explanatory layer.
- One line naming the pain.
- No product claim yet.

### 2. The Transformation

Use for the proof.

- Before and after on the same asset.
- One burnt-orange arrow or underline.
- Label what changed: `SCATTERED NOTES -> CURRENT PROJECT STATE`.
- Do not hide manual steps.

### 3. The Anatomy

Use for education and product clarity.

- One real screenshot.
- Maximum three numbered annotations.
- Each annotation explains why the surface helps the creator resume or decide.
- No feature soup.

### 4. The Return

Use to demonstrate project memory over time.

- Day/date marker on the first capture.
- A genuine delay before the return capture.
- Show what persisted, what changed, and the current next move.
- Best recurring signature for the `project remembers` differentiator.

| Surface            | Package                                    | Opening requirement                                         | Supporting copy                                         |

## HyperFrames Assembly Lane

Use [the BuildOS HyperFrames workflow](../../visual-assets/hyperframes/README.md) after the weekly receipt and story decision are approved.

1. Copy only publish-safe real assets into one composition project.
2. Complete `PROJECT.md` and `STORYBOARD.md` before rendering.
3. Run one draft build; review the five generated QA frames at phone size.
4. Correct material claim, crop, privacy, or legibility problems only; do not generate automatic variants.
5. Run the final silent build. Add licensed music in Instagram for the organic Reel, or create a separately named master using only owned/royalty-free audio.

The first verified implementation is the [Cold Manuscript pilot](../../visual-assets/projects/author-workflow-teardown/cold-manuscript-reel/PROJECT.md). It is an internal real-product example, not a customer receipt, and does not replace the Week 1 requirement to choose the campaign's owned real project.

```text

```

Store publish-ready media in the existing approved marketing asset location used by the publishing workflow; do not commit sensitive raw creator footage by default.

## Permission Record

For every non-DJ project or person:

| Person/project | Participation permission | Recording permission | Screenshot permission | Quote permission | Name/handle permission | Revocation/contact notes |
| -------------- | ------------------------ | -------------------- | --------------------- | ---------------- | ---------------------- | ------------------------ |
|                | yes/no                   | yes/no               | yes/no                | yes/no           | yes/no                 |                          |

Permission to use BuildOS is not permission to publish. Obtain each permission independently.
