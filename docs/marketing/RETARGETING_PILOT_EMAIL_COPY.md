# BuildOS Retargeting Pilot Email Copy

_Concrete email copy for the first founder-led batch_

**Campaign ID:** `buildos-reactivation-founder-pilot-v1`  
**Audience:** Dormant signups from the first pilot batch  
**Tone:** Honest, plainspoken, founder-led  
**Primary themes:** context compounding, clearer thinking, collaborative shared context

---

## Teaching Arc

The sequence should teach three ideas in order:

1. **Touch 1:** BuildOS is worth another look because it is better at turning messy work into reusable context.
2. **Touch 2:** Building context now makes AI more useful later.
3. **Touch 3:** Shared context makes collaborative work stronger because people can move in parallel.

---

## Personalization Rules

Use only light personalization in v1.

- `{{first_name}}`
- optional use-case phrase if you know it
- optional onboarding intent if it improves the line naturally

Do not over-personalize with creepy or stale product data.

---

## Touch 1

**Primary subject:** `You tried BuildOS early. It is different now.`  
**Alternates:**

- `BuildOS was early when you tried it`
- `Quick note from the founder`

**Copy:**

```text
Hi {{first_name}},

You tried BuildOS back when it was still early, and I do not think the
product earned a habit yet.

I am reaching out now because the product is much better at the part that
matters most: helping you build context around real work so the AI can help
more over time instead of starting from zero every session.

That is the big difference.

You can dump messy thinking into BuildOS now, come back later, and keep
refining the same project instead of re-explaining everything again.

If other people are involved, shared context makes it even stronger because
everyone can work in parallel from the same project instead of rebuilding the
plan in meetings, docs, and chat threads.

If you are open to a 5-minute test, try one real brain dump here:
[brain dump link]

Use something you are actually working on, not a fake test.

If it still feels weak, reply and tell me why. I read every reply.

DJ
```

**Why this email exists:**

- acknowledges the earlier product honestly
- introduces context as the core value prop
- plants the collaboration angle without making the whole email about teams
- asks for one small test, not a full re-adoption

---

## Touch 2

**Primary subject:** `Why building context now matters`  
**Alternates:**

- `What changed in BuildOS`
- `Why this is more useful than another AI tab`

**Copy:**

```text
Hi {{first_name}},

Quick thought on why I think current BuildOS is worth another look.

Most AI tools are helpful in the moment.
BuildOS is meant to get more useful after the moment.

Every brain dump becomes context:

- what you are trying to do
- what constraints you have
- what decisions already got made
- what still feels unclear

That matters because AI can only help well later if the context exists now.

This is also where collaboration gets interesting.
If two or three people are moving the same project forward, shared context
means people can work in parallel without constantly asking, "wait, what is
the current plan?"

If you want the fast version, watch this:
[demo link]

If you would rather test it yourself, use this:
[app link]

Suggested test: brain dump one active project today, then come back tomorrow
and ask BuildOS what changed or what should happen next.

DJ
```

**Why this email exists:**

- teaches the "build context now, benefit later" idea explicitly
- sharpens the distinction between BuildOS and stateless AI tools
- explains the collaboration value prop in plain language

---

## Touch 3

**Primary subject:** `One last note on BuildOS`  
**Alternates:**

- `Should I leave you alone on this?`
- `Can I tell you if BuildOS is actually a fit?`

**Copy:**

```text
Hi {{first_name}},

Last note from me on this.

The reason I keep building BuildOS is that I think context is the missing
layer in most AI tools.

If you build context now, the AI can help you more later.
If you share that context with other people, the project can move faster
because everyone is working from the same source of truth.

If you want, reply with one sentence on what you are trying to organize right
now, and whether it is mostly solo or collaborative.

I will tell you honestly whether current BuildOS feels like a fit.

If not, no worries. I will leave it there.

DJ
```

**Why this email exists:**

- closes the loop without pressure
- invites a reply instead of forcing another click
- makes the collaboration value prop more explicit
- gives the founder a clean opening for high-signal conversations

## Optional Personalization Snippets

Use these sparingly in a manual campaign.

### If the user appears solo

Insert after the second paragraph in Touch 3:

```text
Even if you are working solo, the payoff is the same: the AI gets better when
the project has real history to work from.
```

### If the user appears collaborative

Insert after the second paragraph in Touch 3:

```text
Where BuildOS starts to feel especially different is when multiple people are
touching the same project and the system holds the shared context for all of
them.
```

### If the user barely used the product

Insert near the CTA in Touch 1:

```text
You do not need to learn the whole product. Just test one real brain dump.
```

---

## Send Notes

- Keep formatting simple
- Avoid images in Touch 1 unless absolutely necessary
- If you use a demo asset, Touch 2 is the best place for it
- Do not add fake urgency
- Do not add "AI-powered" language to the headline unless you are explaining the mechanism
