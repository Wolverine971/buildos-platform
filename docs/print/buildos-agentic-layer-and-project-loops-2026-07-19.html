The Agentic Layer & Agentic Chat

BuildOS has two AI execution tempos on one project-data backbone: Agentic Chat handles a bounded foreground turn; durable Agent Runs continue in the background. Project Loops are a third, maintenance-oriented producer that sends reviewable work into the AI Inbox.

Foreground

Agentic Chat

A live, streamed LLM/tool loop inside one web request. It is best for conversation, retrieval, quick mutations, and supervising longer work.

Background

Agent Runs

Durable worker jobs with budgets, scoped ops, progress events, steering, pause/resume, results, and optional review-before-commit.

Maintenance

Project Loops

Scheduled or event-triggered project reconciliation. They generate bounded, fresh suggestions and complete-audit follow-ups for human review.

The five rules that make the architecture click

  1. Chat is a supervisor, not the container for all work. A turn may do work inline or dispatch a durable run.
  2. Supabase owns durable truth. Streams and UI cards are views over sessions, turns, runs, events, tools, suggestions, and inbox rows.
  3. Tools are progressively disclosed. The model starts with contextual tools plus discovery; more exact tools materialize as needed.
  4. Writes are concrete and audited. Chat writes execute through validated tool handlers. Agent Runs commit by default or stage when review_required.
  5. Every loop is bounded. Tool rounds, calls, tokens, time, cost, concurrency, deduplication, and stale-run recovery prevent runaway work.

One backbone, distinct executors

Chat and Agent Runs share canonical operations and ontology policy, but they do not share the same lifecycle.

  • Chat: request + SSE + chat tool executor.
  • Agent Run: queue + worker-safe shared op gateway.
  • Project Loop: specialized worker generators.
  • Inbox: denormalized review index over source rows.

State ownership map

Object Durable source of truth What the UI is showing
Conversation chat_sessions, chat_messages Modal transcript; resumed/reloaded from stored messages
One chat turn chat_turn_runs, checkpoints, events, prompt snapshots Streaming text, thinking/tool activity, final outcome
Background work agent_runs, events, signals, tool executions Run Stack, Work Panel, in-chat run dock
Project maintenance project_loop_runs, suggestions, audits Project Review/Audit surfaces and AI Inbox
Human attention inbox_items plus the original source row Pending/deciding/deferred/decided review cards
Project knowledge Ontology project, task, document, goal, event, edge tables Context assembled for chat, runs, loops, and audits

What is deliberately coupled

  • Web and worker use shared types and shared agent operations.
  • Chat-triggered runs link to their parent session/message.
  • Completion can inject a durable assistant message back into chat.
  • Proposals and loop findings converge in the AI Inbox.

What is deliberately independent

  • An Agent Run does not need a chat session.
  • Closing chat does not cancel background work.
  • Manual/scheduled Operatives can dispatch the same run substrate.
  • Loops do not recurse when applying their own accepted mutations.
Architectural consequence: if an interface disconnects, the durable row still matters. Realtime and SSE improve feel; they are not the authority.

Inside one Agentic Chat turn

A turn is not “send the transcript to a model.” It is an admitted, access-checked, context-assembled, observable model/tool loop with a durable finish path. 

 0. MODAL OPENS
    POST /api/agent/v2/prewarm
    -> warm context cache; optionally create session; prepare one-shot prompt row

 1. USER SENDS
    optimistic local message + stream_run_id + client_turn_id
    -> POST /api/agent/v2/stream

 2. ADMIT + AUTHORIZE
    authenticate -> project/daily-brief access -> one-running-turn/session lock
    -> resolve/create chat session -> persist/idempotently identify user turn

 3. BUILD MODEL INPUT
    prepared prompt (preferred) OR 2-minute session cache OR fresh context load
    + recent/compressed history + last-turn continuity + current domain signals
    + contextual direct tools + discovery surface

 4. RUN CONTROL LOOP
    model pass -> text and/or tool calls -> validate/enrich -> execute -> observe
         ^                                                        |
         +---------------- next model pass ------------------------+
    supervisor watches repetition, validation failures, reads, writes, and silence

 5. FINISH
    sanitize final text -> persist assistant + tool trace + turn outcome
    -> emit timing + last_turn_context + done -> reconcile session agent_state

 6. CLEANUP / RECOVERY
    batch turn events; write metrics/snapshots; preserve interruption reason;
    checkpoint a user-question stop so the next turn can resume with structured state
300s

Maximum Vercel stream duration. Long work should become an Agent Run rather than stretching the request.

12 / 9

Default gateway tool-round cap; reduced to 9 when prompt context is near or over its budget.

40

Default tool-call ceiling. Repetition and supervisor synthesis guards often stop the loop earlier.

Context composition

Layer What it answers
Scope Global, project/audit/forecast, ontology focus, or daily brief?
Working set Which tasks, documents, events, goals, and summary matter now?
History Recent raw turns plus compressed earlier context
Continuity Which exact entities and unfinished intent survived the last turn?
Runtime Current tool results, cancellation, cache freshness, domain signals

Current history/cache defaults

  • History lookback: last 10 messages.
  • Compression threshold: 8 messages.
  • Compressed tail retained: 4 messages.
  • Summary cap: 420 characters.
  • Per-message compressed cap: 1,200 characters.
  • Session context cache TTL: 2 minutes.
  • Prepared prompts: server-owned, nonce-protected, one-shot.

Stream contract: what the browser sees

agent_state -> session -> context_usage -> repeated text_delta / tool_call / tool_result -> optional context_shift -> timing -> last_turn_context -> done (or terminal error). The client uses run IDs to reject stale events.

Concurrency rule: the database admits one running turn per chat session. A recent turn blocks a second; a stale detached turn is cancelled before a new row attempts to acquire the lock.

The model-facing rule

Capabilities orient. Skills teach. Tool search finds. Tool schema specifies. Direct tools execute.

Simple reads can skip skill loading. Multi-step, stateful, or risky work should load workflow guidance before mutation.

What the execution layer does before a call runs

  • Maps aliases to the canonical operation and direct tool.
  • Injects context-derived project_id when valid.
  • Normalizes UUID fields, defaults, and supported argument aliases.
  • Checks user/project/entity scope and validates the exact schema.
  • Applies timeouts and the active abort signal.
  • Returns affected entities and a durable tool-execution receipt.

Deterministic runtime brakes

Failure mode Runtime response Why it matters
Repeated identical reads Within-turn read memo + novelty/repetition tracking Prevents a “search forever” loop
Missing/invalid arguments Bounded validation repairs; then ask user or stop No guessing IDs or fields
Pending write near limit One restricted write-only pass when safe Stops reads from consuming the entire turn
Duplicate successful write Stable in-turn write dedup key; skip repeat execution A retry cannot double-create/update
Too many read rounds / calls Supervisor forces no-tool synthesis User gets an answer instead of a timeout
Output hits length cap Up to two bounded continuation passes Does not present a cut-off answer as complete
User stops or supersedes Abort plus persisted interruption reason Partial output remains auditable
Supervisor asks user Checkpoint digest + resume context Next turn continues structurally, not from guesswork

Write integrity

  • Reads are re-derivable; writes persist incrementally for crash recovery.
  • Successful writes are tracked from executor receipts, not model claims.
  • Destructive or under-specified operations do not get autonomous recovery.
  • Finalization checks requested intent against actual tool outcomes.

Observability

  • Turn run, ordered turn events, tool trace, and timing metrics.
  • Prompt snapshots and cost breakdowns for evaluation/debugging.
  • Context-usage tracking has separate UI and orchestration budgets.
  • Search telemetry records result counts and zero-result outcomes.
The design center: keep the model-facing concepts small while the runtime spine—state, guardrails, memory, events, persistence, and observability—does the enforcement.

Durable Agent Runs: the background action loop

An Agent Run is a first-class work object—not a long chat request. It can be triggered by chat, a person, a schedule, or an event; it survives UI closure and reports back through durable state.

Hard platform boundaries

  • 3 active runs per user at dispatch.
  • Depth cap 1: delegated runs cannot delegate again.
  • Default 20 op calls in the worker loop.
  • Standard: balanced model, 5-minute default wall clock.
  • Deep: powerful/high-reasoning model, 10-minute default.
  • Budget fields can cap wall clock, tokens, tool calls, and cost.
  • Deep runs are capped at $1 dispatch budget; global run budget max is $10.

Lifecycle and user control

Status Meaning Next transition
queued Durable row exists; worker has not claimed it running or cancel/fail
running Worker action loop is active steer, pause, result, cancel, budget stop
paused Checkpointed; worker slot released resume re-enqueues a continuation
needs_input Agent cannot proceed without an answer answer signal + continuation job
proposal_ready Review run staged one or more writes approve/reject Change Set
partial Useful result, but not fully complete continue if open questions exist
completed / failed / cancelled Terminal outcome with result/error/metrics inspect, retry, or start a new run

Direct commit

Default for read_write. The operation gateway applies writes, and entity touches come from execution telemetry.

Review first

review_required turns write ops into proposed changes. The run ends at proposal_ready until a human commits the Change Set.

Steer mid-flight

Signals are drained between iterations. A steer becomes transcript context; pause checkpoints; cancel finalizes gracefully.

Conditional capabilities: calendar operations appear only when the worker has Google credentials and the user has stored tokens. Review-mode calendar runs expose reads only. Web research treats fetched content as untrusted evidence and uses durable paid-tool cost reservations.

Project Loops

Project Loops are BuildOS’s self-maintenance system. They periodically or eventually re-read a project, generate bounded review items, keep only fresh findings, and route the few highest-priority asks through the AI Inbox.

What a light loop looks for

  • Document organization: misplaced or structurally weak docs.
  • Outdated documents: stale content worth flagging.
  • Project drift: evidence that execution diverged from intent.
  • Task conflicts: duplicate, contradictory, or blocking task pairs.

What a complete audit adds

  • Project thesis and delivery confidence.
  • Dimension ratings, top findings, risks, and open questions.
  • Evidence-linked recommendations and a durable report.
  • Up to eight individual, actionable follow-up suggestions.
The key mental model: source tables are the candidate/history ledger; inbox_items is the attention surface. Rotation, TTLs, and the per-project budget decide what deserves to interrupt the user now.

Current code status

Web and worker feature switches are now always-on compatibility exports. The worker registers the project-loop processor, checks light loops hourly against each owner’s local midnight window, scans complete audits daily, and runs stale lifecycle repair every 30 minutes.

Burst source weights

  • Project update: 5
  • Goal or milestone: 5
  • Risk: 4

The light loop: reconcile, suppress, rotate

The light loop is a bounded reconciliation pass. Its output is not a free-form report: it is a brief plus typed suggestions with evidence, previews, operations, undo data, risk, and freshness metadata. 

 1  CLAIM RUN         queued -> running (status-fenced; duplicate workers lose)
 2  LOAD CONTEXT      active project + graph + bounded docs/tasks + prior decisions
 3  BUILD BRIEF       LLM brief; heuristic fallback if cost path is unavailable
 4  RUN GENERATORS    doc organization -> outdated docs -> drift -> task conflicts
 5  CAP + SUPPRESS    max 25; semantic keys; 60-day open/decided lookback
 6  STAMP FRESHNESS   fingerprint only the task/doc entities an operation mutates
 7  INSERT + INDEX    project_suggestions -> inbox_items
 8  ROTATE            reconfirm matching old findings; supersede unconfirmed >72h
 9  ADMIT ATTENTION   rank risk then freshness; pending top 3; defer overflow
 10 FINISH RUN        waiting_review when new suggestions exist; else completed

The four generators

Kind Question Typical result Safety shape
doc_org Is a document misplaced or the tree confusing? Document move proposal Reversible operation + prior parent for undo
doc_outdated Does a document conflict with newer project state? Outdated metadata flag Scoped doc fingerprint + reversible props update
drift Is the project diverging from its goal or plan? Informational finding No direct mutation; address/discuss/dismiss
task_conflict Are two tasks duplicate, contradictory, or blocking? Conflict metadata proposal Deterministic candidate pairs + IDs validated

Duplicate suppression

Keys are based on operation targets: order-independent task pairs, document ID sets, evidence IDs, or a normalized-title fallback. They are kind-namespaced and checked against open/recently decided suggestions for 60 days plus duplicates in the current run.

Scoped freshness

Executable proposals hash only the entities they will mutate. At approval, the web recomputes that subset. If it changed, the suggestion becomes superseded instead of replaying stale operations.

Rotation: refresh, do not append forever

  1. A new matching finding counts as reconfirmation—even if insertion was suppressed.
  2. Reconfirmation bumps updated_at and extends review expiry.
  3. An unconfirmed prior light finding survives a 72-hour grace window.
  4. After grace, it is superseded as “not reconfirmed by latest run.”
  5. Items the user is deciding/snoozing and generator kinds skipped by cost are protected.

Cost behavior

The light-run cap is $0.35. It is checked before the brief and each generator, so later generators are skipped after the threshold is reached. A single in-flight call can still take the measured total past the cap; the cap is a between-call gate, not a provider-side hard reservation.

Complete Project Audit: the slower strategic loop

A complete audit is evidence-grounded synthesis over a mature-enough project. It keeps a deterministic scaffold as a fallback, then uses an LLM to sharpen the thesis, confidence, findings, dimensions, risks, questions, and recommendations.

Automated baseline

An automated audit requires all of:

  • Project active or planning.
  • At least 7 days old.
  • Activity on at least 2 distinct days.
  • A goal or an 80+ word project description.
  • At least 3 of 5 content threshold groups.

Manual audits bypass these baseline gates, but still respect active-audit and queue ownership controls.

Content threshold groups (need 3)

Group Threshold
Documents 5 active docs or one 1,500+ word substantial doc
Tasks 5 non-deleted tasks
Direction At least 2 goals + milestones
Commitment Any project/task/milestone/event date
Graph size At least 10 total entities

Automated trigger logic

Size class Scheduled rule Burst rule
Small eligible Manual only 72h score >=10 and at least 25% of entities touched
Medium 14d cadence; at least 3 changed entities or a major change 72h score >=16 or 12 changed entities in 7d
Large 14d cadence; at least 6 changed entities or a major change 7d score >=24 or at least 15% of entities touched
Strategic Same scheduled gate as large Large rule or at least 2 major changes in 72h

   4. synthesize with evidence-constrained LLM (fallback keeps scaffold)

Audit memory

Uses prior audit recommendation decisions and light-loop decisions to avoid repeating already reviewed asks.

Cost

LLM synthesis has a $0.35 soft cap. Failure logs a warning and falls back to the deterministic audit packet.

Inbox shape

The parent report remains project history; individual actionable children enter the Inbox and share its budget.

Quiet period: a qualifying burst waits until two hours after the last major mutation, then a scheduled trigger-evaluation job rechecks the project. This avoids auditing while a restructuring burst is still underway.

Attention and expiry policy

Source TTL
Light-loop finding 7 days
Audit recommendation 14 days
Audit parent 14 days
Agent Run proposal 14 days
Calendar suggestion 7 days or event window +48h, whichever is sooner

Per project, the top three project-member items are pending; overflow is deferred. Ranking is risk tier, then freshness. A freed slot promotes the next deferred row.

Operational failure controls

Duplicate execution

Queue dedup + returned-metadata ownership + status-fenced claims prevent two workers from inserting the same run’s suggestions.

Stalled lifecycle

Every 30 minutes: fail running rows older than 1h, queued rows older than 6h, and finish review runs whose children are all decided.

Failed parent

Pending children from a failed loop are superseded and re-indexed so the Inbox does not present orphan work.

Recursive trigger

Accepted suggestion replay carries structured review context that suppresses a new mutation-burst loop.

Stale proposal

A changed scoped fingerprint supersedes the item with zero applied operations and asks for a fresh review.

Index drift

Read-time reconciliation and source backfill re-sync Inbox rows; shared admission policy prevents backfill from bypassing the budget.

Known edges to keep in mind

  • Complete-audit daily scanning is worker-clock, not per-user local time.
  • critical_change is modeled but has no current producer.
  • Informational/no-op findings rely on rotation and memory, not entity fingerprints.
  • Generator quality is still empirical: dismissal/repeat telemetry matters.
  • Source-state stalls can need source-specific repair even when the index is correct.
