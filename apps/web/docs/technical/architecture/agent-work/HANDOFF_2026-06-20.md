## 1. Current calendar status (initial Waves 5–6 shipped)

**Goal:** let an Agent Run read calendar context and, in direct-commit runs, create/update/delete calendar events and set project calendars. There are 7 `cal.*` ops total: 3 reads (`cal.event.list`, `cal.event.get`, `cal.project.get`) and 4 writes (`cal.event.create`, `cal.event.update`, `cal.event.delete`, `cal.project.set`). The initial worker implementation is now in `packages/shared-agent-ops/src/calendar/agent-run-calendar-port.ts`.

### What shipped

- `createAgentRunCalendarPort({ admin, userId, credentials? })` is exported from `@buildos/shared-agent-ops`.
- The port has all 7 `CalendarPort` methods and uses stored encrypted `user_calendar_tokens` plus backend Google OAuth credentials.
- The worker creates the port only when `PRIVATE_GOOGLE_CLIENT_ID`, `PRIVATE_GOOGLE_CLIENT_SECRET`, `PRIVATE_CALENDAR_TOKEN_ENCRYPTION_KEY`, and a user token row are present.
- `buildAgentRunOpCatalog({ scope, mutationMode, calendar })` exposes `cal.*` only when that port exists; stage/review mode exposes calendar reads only.
- `onto_event_sync.calendar_id` is treated as `project_calendars.id`; raw Google calendar IDs are used only for Google API calls.
- Headless checks are green. A live create/read/delete smoke is still required.

### Historical seam

Wave 7 left calendar **fully wired in the package** behind a port. The worker now supplies an initial implementation of that port when the runtime has the required env + user token state.

- The package's gateway handler map (`op-execution-gateway.ts`) **keeps all 7 `cal.*` handlers**. They call `context.calendar` — a `CalendarPort`:
    ```ts
    export interface CalendarPort {
    	listCalendarEvents(args): Promise<unknown>;
    	getCalendarEventDetails(args): Promise<unknown>;
    	createCalendarEvent(args): Promise<unknown>;
    	updateCalendarEvent(args): Promise<unknown>;
    	deleteCalendarEvent(args): Promise<unknown>;
    	getProjectCalendar(args): Promise<unknown>;
    	setProjectCalendar(args): Promise<unknown>;
    }
    ```
- **Web** supplies one: `createCalendarPort(params)` in `external-tool-gateway.ts` wraps the concrete `CalendarExecutor`.
- **Worker** supplies one: `createAgentRunCalendarPort(params)` in `@buildos/shared-agent-ops` talks to Supabase + Google Calendar APIs without SvelteKit imports.
- The non-calendar write catalog remains calendar-free by design:
    ```ts
    // op-execution-gateway.ts:6241
    export const AGENT_OP_GATEWAY_WRITE_CATALOG = Object.freeze(
    	Object.keys(EXTERNAL_OP_HANDLERS).filter((op) => isWriteOp(op) && !op.startsWith('cal.'))
    );
    ```

So the current job is no longer "add any worker port"; it is **live-smoke the initial port**, then decide whether to consolidate more of the web calendar executor into the package.

### Decision: the worker should use stored user tokens + backend app creds

The worker does not need a fresh Google login. BuildOS already stores user OAuth tokens in `user_calendar_tokens`; the worker should decrypt and use those tokens with the same Google OAuth client credentials the web backend uses. The app credentials live in worker/server environment variables, not in the DB; user access/refresh tokens stay encrypted in the DB.

This matches the current Google OAuth model: a backend web-server app can securely store confidential app credentials, request offline access, and refresh user access tokens after the user leaves. It also matches our Supabase posture: the worker runs with backend/service credentials, so it may read encrypted token rows, but that power stays out of browser/client code.

### R4 status: GoogleOAuthService no longer imports `$env/static/private`

`apps/web/src/lib/services/google-oauth-service.ts` used to import Google creds from SvelteKit `$env`:

```ts
import { PRIVATE_GOOGLE_CLIENT_ID, PRIVATE_GOOGLE_CLIENT_SECRET } from '$env/static/private'; // 9 usages
```

The worker can't load `$env`, so this refactor is now complete for `GoogleOAuthService`: it accepts constructor credentials and falls back to backend `process.env`:

```ts
constructor(supabase, credentials?) {
  this.clientId = credentials?.clientId ?? process.env.PRIVATE_GOOGLE_CLIENT_ID!;
  this.clientSecret = credentials?.clientSecret ?? process.env.PRIVATE_GOOGLE_CLIENT_SECRET!;
}
```

Keep this invariant during the carve: no package code may import `$env/static/private`. `apps/web/src/lib/utils/google-oauth.ts` remains a web utility; do not pull it into the worker package unless it receives the same treatment.

### Wire-up status

1. Worker builds a `CalendarPort` from the package factory when Google env vars + user tokens exist.
2. `AgentOpContext.calendar` flows to gateway read/write execution.
3. Keep separate catalogs:
    - `AGENT_OP_GATEWAY_CALENDAR_READ_CATALOG` = the 3 calendar reads.
    - `AGENT_OP_GATEWAY_CALENDAR_WRITE_CATALOG` = the 4 calendar writes.
    - `AGENT_OP_WRITE_CATALOG` remains non-calendar because non-calendar writes are stage+commit-safe without a calendar capability.
4. Runner uses `buildAgentRunOpCatalog({ scope, mutationMode, calendar })` instead of raw catalog merging:
    - no `CalendarPort` → no `cal.*` tools.
    - `CalendarPort` + commit mode → calendar reads + writes.
    - `CalendarPort` + stage/review mode → calendar reads only; calendar writes are hidden and `executeAgentOp` rejects them if called directly.
5. **Phase 4 staging for calendar:** do **not** silently commit calendar writes during review mode. V1 blocks calendar writes in `mutationMode:'stage'`. Full staging is a follow-up that needs a faithful external intent, idempotent Google event creation, and commit-time replay semantics.
6. `changeSetCoverage.test.ts` and `calendarInRuns.endstate.test.ts` now cover this split: `cal.*` are capability-supported but still not part of the always-on non-calendar write catalog.

## 4. Invariants & landmines (don't relearn these)

1. **Worker can't import `apps/web`** — that's why the op + gateway + (next) calendar logic lives in `@buildos/shared-agent-ops`. Package src must have **ZERO** `$lib`/`$env`/`$app` imports (grep-verify after every edit).
2. **Rebuild the package before web/worker typecheck/tests** — stale `dist/` gives false results. The #1 time-waster.
3. **For Wave 5/6, run the full `pnpm --filter @buildos/web build`, not just `check`** — a missed `$env/static/private` line passes typecheck but fails the Vercel build. ProjectCalendar `Response`→object (R5) breaks `.json()` callers; carve carefully.
4. **One write path:** all writes (direct + Change Set commits) go through `runGatewayWriteOp` → the gateway handler map. Don't add a second path; add ops to the map / wire the port.
5. **Staging keeps the LLM unaware** (02 §3): a staged write returns synthetic success; `review_required` (not the model) decides commit vs stage. Don't tell the runner prompt about staging.
6. **`commitChangeSet` claims the run** (`proposal_ready`→`running`) before applying — it's the only guard against concurrent double-apply (writes have no idempotency key). Keep it.
7. **`supabase_realtime` publication is dashboard-managed** — `postgres_changes` on a new table silently delivers nothing until added; the Run Stack polls as the reliable path.
8. Older landmines (file-write `\u` mangling, `add_queue_job` RPC signature, enum-cast on `.in()/.eq()`) — `HANDOFF_2026-06-18.md` §6.
