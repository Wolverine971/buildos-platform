## 1. What this feature is (one paragraph)

An **Agent Run** is a durable background task that runs in the **worker** (not the web request), executes BuildOS ops on its own, streams events, and reports a structured result. The agentic chat _supervises_ runs rather than doing all work inline. Two locked product decisions: **(a) mutations commit directly by default; review-before-commit is per-run opt-in** (`review: true`), and **(b) runs are steerable** (cancel, steer, pause, resume, and the `needs_input` answer flow are all built as of 2026-06-18).

### Hardening pass — ✅ COMPLETE 2026-06-19

- ✅ **Project access enforcement:** worker read ops now fail closed unless the run user has actor-specific project access; project-scoped runs cannot read another project's graph/tree, and `onto.project.list` is filtered to the run project.
- ✅ **Continuation safety:** resume/answer routes atomically claim `paused` / `needs_input` runs, use stable queue dedup keys, include explicit `continuation_from`, and roll back both run status and inserted signals if enqueue fails.
- ✅ **Duplicate/stale worker protection:** the worker conditionally claims runs by current status and skips stale jobs instead of unconditionally setting runs to `running`.

- ✅ **Budget/result tuning:** `max_tokens` is enforced; `max_tool_calls` allows one final no-more-tools `submit_result` turn; `completed` with `open_questions` is normalized to `needs_input`.
- ✅ **Verification:** shared package builds/typechecks, worker typecheck, web check, worker targeted tests (`22/22`), web agent-call/ontology guardrail tests (`205/205`), and worker lint (0 errors, existing warnings only).

## 3. Key decisions & invariants (do not relitigate without reason)

1. **The worker cannot import `apps/web`** (separate package, no `$lib`). That's _why_ the op layer was extracted into `@buildos/shared-agent-ops`, importable by both web and worker.
2. **Extraction tactic = move + re-export shim.** Moved file's contents go into the package; a one-line `export * from '@buildos/shared-agent-ops'` shim stays at the old `$lib/...` path so importers don't churn. **Shims stay until a dedicated cleanup pass** (out of scope).
3. **Barrel collisions → subpath exports.** Some modules export colliding symbol names (`onto`/`onto-api` both define `DocTreeNode`/`DocumentChildren`; `project-graph.types` exports `EntityKind`). Those are NOT in the flat barrel — they're exposed via subpath (`@buildos/shared-agent-ops/ontology/onto`, etc.). **To add another:** add the file to `entry` in `packages/shared-agent-ops/tsup.config.ts` + an `exports` entry in its `package.json`, then point the shim at the subpath.
4. **Runner uses a JSON action-loop, not native tool-calling** — the worker's `SmartLLMService.getJSONResponse` is the proven path and robust across BuildOS's model range. The LLM emits `{action:'call_op'|'submit_result', ...}` each turn.
5. **Read-first.** `executeAgentOp` runs read ops only; write ops return `UNSUPPORTED`. Writes need the gateway carve (§5).
6. **Telemetry = dedicated `agent_tool_executions`** (NOT a nullable `session_id` on `chat_tool_executions`) — `createServiceClient`/`process.env` is **not proven in the web runtime**, so don't assume it.
7. **`security-event-logger` was deferred to Wave 7** (its web fallback uses `$env`-based `createAdminSupabaseClient`; the package version must require `options.supabase` and the web shim must inject admin).

## 6. Landmines (things that cost me time — save yourself)

- **The file-write channel mangles `\u` escapes** — ` ` in written content collapses to a literal control char. For control-char regexes, use a charCode filter or write via a Node heredoc with doubled backslashes. (Bit me in `search-filter.ts`.)
- **Rebuild the package before web/worker typecheck or tests** — they consume `dist/`, not `src/`. Forgetting this gives stale/false results.
- **`gen:types` needs `SUPABASE_ACCESS_TOKEN`** (now in `.env`) or `supabase login`. It silently keeps stale types otherwise — verify `database.types.ts` actually changed (mtime/grep) after running.
- **`add_queue_job` RPC signature**: `p_user_id, p_job_type, p_metadata, p_priority, p_scheduled_for, p_dedup_key` (see the agent-runs POST route / tree-agent route).
- **Supabase `.in()/.eq()` on enum columns** want the literal union type, not `string` — import `AgentRunStatus` and cast (bit me in the list route).
- **The working tree has pre-existing dirty files** unrelated to this work (agentic-chat-lite, agentic-chat-v2, etc., dirty since session start). Don't attribute them to Agent Work; don't revert them.
- **Inlined `ServerTiming`** in `ontology-projects.service.ts` (package copy) is a minimal structural interface — don't try to re-import the SvelteKit Server-Timing helper into the package.
- **The `supabase_realtime` publication is dashboard-managed, NOT in migrations** — no `ALTER PUBLICATION` exists anywhere (yet `queue_jobs` realtime works). So `postgres_changes` on a new table (`agent_runs`) silently delivers nothing even when the channel reports `SUBSCRIBED`, until the table is added to the publication. That's why the Run Stack's polling fallback is unconditional (never gated on channel "health"). The shipped migration adds `agent_runs` to the publication; apply it (or add it via the dashboard) for instant updates.
