## 2. Wave 7 ‚Äî write ops (SHIPPED)

Runs execute BuildOS write ops in-process through the **same gateway handler map the live chat uses** ‚Äî one implementation, not two.

- **The carve:** `apps/web/.../external-tool-gateway.ts` (was 6,603 lines) ‚Üí a 572-line web **adapter** (discovery + port adapters + registry wiring). The op-execution core (handler map, `executeGatewayOp`, `buildExternalGatewayRegistry`) moved to **`packages/shared-agent-ops/src/gateway/op-execution-gateway.ts`**.
- **Dependency-inverted registry:** `buildExternalGatewayRegistry(scope, registryOps, registryVersion)` and `executeGatewayOp({‚Ä¶, registryOps, registryVersion})` no longer call `getToolRegistry()`; web injects the registry ops/version. Package src has **ZERO** `$lib`/`$env`/`tool-registry`/`CHAT_TOOL_DEFINITIONS` imports (grep-verified).
- **Ports for the two web-coupled deps:** `CalendarPort` and `TaskSyncPort` on `ToolExecutionContext`. Web supplies concrete adapters. The worker now supplies an initial package-native `CalendarPort` when env + user tokens are available; task-sync is still skipped in worker runs.
- **Worker op path:** `runGatewayReadOp`/`runGatewayWriteOp` (lean: validate + invoke the handler directly, NO external `agent_call_tool_executions` audit ‚Äî the runner records its own `agent_tool_executions`). Wired into **`packages/shared-agent-ops/src/gateway/op-execution.ts`** `executeAgentOp`: allowed-op gate ‚Üí run-project fence ‚Üí read/write dispatch. Catalogs = non-calendar read/write catalogs plus separate calendar read/write catalogs that the runtime catalog builder includes only when a `CalendarPort` is present.
- **Support carves (worker-safe):** `security-event-logger` (R3 DI cut), `write-audit.service`, `agent-call-project-activity.service` ‚Üí package, web shims at the old paths.
- **Guardrail:** `external-tool-gateway.test.ts` (37 cases) stayed green throughout ‚Äî behavior-preserving carve. Live smoke: `createTask`/`updateTask` create/update real rows; cross-project & `read_only` writes ‚Üí `FORBIDDEN`.

## 3. Phase 4 ‚Äî staged mutations (SHIPPED)

Opt-in review-before-commit. **Default is still direct commit** (`mutationMode='commit'`); only a run dispatched with `review` stages. Design + open questions in [`02-STAGED-MUTATIONS.md`](./02-STAGED-MUTATIONS.md).

- **Schema/contracts already existed** (no migration): `agent_runs.change_set` (jsonb), `review_required`, `proposal_ready` status, `agent_tool_executions.{mutation_mode,proposed_change_id}`, and the `ChangeSet`/`ProposedChange` types in `agent-work.types.ts`. Dispatch already rejected `review`+`read_only`.
- **Stage path** (`op-execution-gateway.ts` `stageGatewayWriteOp` + `op-execution.ts` `executeWriteOp` stage branch): when `ctx.mutationMode==='stage'`, a write op validates args, derives action/entity, fetches a compact `before` snapshot for update/delete, and returns a `ProposedChange` (no mutation, synthetic success so the LLM loop continues unaware).
- **Runner** (`agentRunWorker.ts`): `mutationMode = review_required && read_write ? 'stage' : 'commit'`. Staged changes accumulate (runner assigns `randomUUID` ids), finalize as **`proposal_ready`** with a pending `ChangeSet` on `agent_runs.change_set` + a `run.proposal` event. A review run that staged nothing finalizes normally.
- **Commit** (`packages/shared-agent-ops/src/gateway/change-set.ts` `commitChangeSet` + `POST /api/agent-runs/[id]/commit`): applies approved fresh changes via the same `runGatewayWriteOp`, records `applied_entity_id`/`error` per change, promotes applied entities into `entities_touched`, flips run ‚Üí `completed`/`partial`. **Atomically claims the run (`proposal_ready`‚Üí`running`) before applying** so concurrent commits can't double-apply (live-tested). Per-change `allowed_ops` re-check. Known ontology update/delete changes re-fetch the current row and fail stale proposals instead of clobbering user edits made after review generation.
- **UI** (`ChangeSetReview.svelte`): per-change approve/reject + before‚Üíafter diff + Apply, in `AgentRunModalContent` (and the Work Panel detail view) when `status='proposal_ready'`. Work Panel list already flags `proposal_ready` as "needs you".
- **Chat** (approval flow #2): `delegate_task` gained a `review` flag (read_write only); new `commit_change_set` orchestrator tool ‚Üí `UtilityExecutor.commitChangeSet` ‚Üí the shared `commitChangeSet`. Registry op `util.agent.commit_changes`, surfaced in `global_basic`+`project_basic`.
- **Coverage test** (`apps/worker/tests/changeSetCoverage.test.ts`): every `BUILDOS_AGENT_WRITE_OPS` op is stage+commit-supported (non-calendar, has a handler) or explicitly excluded (`cal.*`). Fails loudly if a future write op slips through.
- **Live-confirmed:** stage create ‚Üí no mutation; full stage‚Üícommit roundtrip (approve-1/reject-1 ‚Üí only approved row created); concurrent commit ‚Üí 1 `CONFLICT` + 1 success, exactly one row.

### Phase 4 known v1 limitations (documented, not bugs)

- Drift detection is limited to staged update/delete changes on known ontology tables. Relationships, cross-entity dependencies, and external resources still need richer conflict semantics.
- `ChangeSet` status `partially_applied` is used even when **nothing** applied (the enum has no `failed`); the **run** status is correctly `partial` in that case.
- Calendar writes can't be staged in runs. Calendar reads can be exposed once a worker `CalendarPort` exists; calendar writes are direct-commit only until full external-side-effect staging exists, and are blocked in review mode.
- Auto-approve Operative policy (approval flow #3) is **Phase 6**, not built.

## 6. Key invariants & landmines (don't relearn these)

1. **Worker can't import `apps/web`** ‚Äî that's why the op layer + gateway carve live in `@buildos/shared-agent-ops`.
2. **Package src must stay worker-safe** ‚Äî ZERO `$lib`/`$env`/`tool-registry`/`CHAT_TOOL_DEFINITIONS`/`skill-load` imports (grep-verify after any gateway edit).
3. **Rebuild the package before web/worker typecheck/tests** ‚Äî they consume `dist/`, not `src/`. Stale `dist/` gives false results.
4. **One write path:** both direct commits and Change Set commits go through `runGatewayWriteOp` ‚Üí the gateway handler map. Don't add a second write path; add ops to the handler map.
5. **Staging keeps the LLM unaware** (02 ¬ß3): a staged write returns synthetic success; the runner ‚Äî not the model ‚Äî decides commit vs stage from `review_required`. Don't add a "you are staging" instruction to the runner prompt.
6. **`commitChangeSet` claims the run** (`proposal_ready`‚Üí`running`) before applying ‚Äî keep that guard; it's the only thing preventing concurrent double-apply (writes have no idempotency key).
7. **Telemetry is keyed by `agent_run_id`** (`agent_tool_executions`), never a chat session; `mutation_mode`/`proposed_change_id` distinguish staged vs committed.
8. **`supabase_realtime` publication is dashboard-managed** ‚Äî `postgres_changes` on a new table silently delivers nothing until added; that's why the Run Stack polls. Migrations `‚Ä¶618000000`/`‚Ä¶618010000` add `agent_runs`/`chat_messages` ‚Äî apply them for instant realtime.
9. Older landmines (file-write `\u` mangling, `add_queue_job` RPC signature, enum-cast on `.in()/.eq()`, inlined `ServerTiming`) are in `HANDOFF_2026-06-18.md` ¬ß6.
