# BuildOS AI Agent Editorial Roadmap

## Goal

Turn the capability-system idea into a practical publishing plan that:

- gives readers a clear starter path
- scales into a coherent library over time
- supports both blog production and social-media distribution
- positions BuildOS as the best integrated implementation without making the content feel closed

This roadmap is the execution layer for the strategy docs.

---

## Reassessment

The plan should not start by publishing random skills.

It should start by teaching readers the system in the right order:

1. what skills, tools, plugins, and stacks are
2. how to start with a simple, high-value integration path
3. how to progress into more capable workflows
4. why BuildOS is the layer that makes the whole thing durable

That means the right first move is a **starter path**, not just a backlog.

## Recommended Starter Path

The best starter path is:

- broad enough to attract interest
- practical enough to feel immediately useful
- simple enough to explain cleanly
- close enough to BuildOS that the differentiation is obvious

Start with **Google Workspace for agents**.

Why:

- everyone understands the use case

- it gives you a natural bridge into BuildOS daily briefs, project context, and execution

## Parallel Tracks To Add Early

This should not be only an integrations roadmap.

You also want BuildOS to become a go-to source for higher-level AI operating judgment. That means running parallel tracks alongside the provider clusters.

### Track A: Research Skills

This track teaches agents how to investigate the world well.

Core ideas to publish:

- lay of the land first, specifics second
- compare search tools by use case, not by hype
- for recent topics, research first before trusting the base model
- AI accuracy generally degrades the more recent the topic is
- post-training and safety behavior can degrade performance on sensitive topics
- triangulate sources, dates, and claims before acting

Priority articles:

1. `How to Do Smart AI Research`
2. `When To Research First: Why Recent Topics Break Naive AI Workflows`
3. `Tavily vs Perplexity vs Brave vs Building Your Own Browser Research Stack`
4. `When To Use Search APIs vs Puppeteer`
5. `How Safety Training Changes AI Behavior On Sensitive Topics`

### Track B: Opinionated Calendar Operations

Because BuildOS uses Google Calendar directly, this track should be more opinionated than generic integration content.

Core ideas to publish:

- search before create
- avoid duplicate events
- define event identity before mutating
- handle recurring events carefully
- scope searches by time window, participants, and event text
- update existing events when confidence is high
- ask for confirmation or search again when confidence is low

Priority articles:

1. `Google Calendar For AI Agents: Search Before You Create`
2. `How To Keep AI Agents From Creating Duplicate Calendar Events`
3. `How AI Agents Should Search Calendars By Time, Title, And Context`
4. `Recurring Events, Event IDs, And Other Calendar Edge Cases`

### Track C: Marketing And Brand Operations

This track teaches softer strategic skills that still need operational rigor.

Core ideas to publish:

- brand foundation before promotion
- brand strategy before content volume
- prioritization before automation
- intelligent promotion rather than spammy posting
- using agents to support editorial systems, not replace judgment

Priority articles:

1. `How To Build A Brand Strategy An AI Agent Can Actually Use`
2. `Marketing Skills For AI Agents: Positioning, Prioritization, And Promotion`
3. `How To Set Up An Agent To Promote Your Work Without Sounding Generic`
4. `From Brand Foundation To Content System: An AI Marketing Operating Model`
5. `How BuildOS Can Turn Marketing Activity Into Durable Strategic Context`

### Track D: Agent Harness And Ecosystem Design

This is where gstack and OpenClaw become especially valuable as reference points.

Core ideas to publish:

- how skill systems shape agent behavior
- how tool surfaces should be narrow and typed
- how plugins package capability
- how agents should progressively discover tools instead of loading giant static catalogs

Priority articles:

1. `What gstack Gets Right About Stackable Agent Skills`
2. `What OpenClaw Suggests About External Agent Capability Surfaces`
3. `Why Agents Need Skills, Tools, Plugins, And Typed Gateways`
